# My goal and design philosophy

First and foremost, my priority for this project was control and precision.

I have made this project to be used by large companies who want velocity in their day to day projects, and precision to ensure that the AI does not hallucinate, and to do exactly what's being asked. For providing control, the user can revise, modify, regenerate and approve/deny at every single step of this project, from PRD generation to sprint backlogs.

## Context problem

Generating a PRD requires "context"; which a user may or may not provide in full. If this context about the application / service / product that they are designing is not complete and understood properly by the team OR the AI, it can lead to hallucinations and degradation of the generated PRD document. A solid PRD means a solid project with fewer bugs.

To ensure precision, I thought of a "detailed mode" which will take a simple user requirement like "generate a chatbot for SCU for students to get answers for most frequent questions" and run it through a classifier to tag topics, intention and context to this simple prompt.

This classifier has many params which help provide further context for our PRD, as well as the next step.

The next step would be for an LLM to judge whether we need more context with respect to the requirement prompt the user provided + category it belongs to. If yes, then the user will be presented with a series of questions to further refine their requirement, and remove ambiguity within their idea. The responses are multi-select and pregenerated by AI for convenience again, but users also have control over what they want their sub-requirements to be and customize everything, fleshing out the requirement in detail.

## Generation of a modular PRD

To ask AI to generate a long document is foolish, because the AI will lose the context we gathered with so much care and hallucinate right in the middle of it.

1. To solve this, PRD generation is modular with sections divided in several sections. Independent of each other but sharing compacted context.
2. Web search yielding market research in real time using 4 separate LLM calls rather than choosing AI to make up statistics and metrics
3. PRD document strictly follows a template we have provided to ensure consistency across generations.

Explanations:

1. Part 1 (gen)     → Overview, problem, users, goals, scope
   Research block   → Web-searched evidence (4 subsections)
   Part 2 (gen)     → Requirements, NFRs, compliance tables
   Part 3 (gen)     → Rollout, risks, acceptance criteria

2. Research categories are: useCases, metrics, scope, compliance

3. The template is a mix of Atlassian template, ISO 25010 and draft templates used by Amazon. In the end I ended up truncating most of the PRD template because it was redundant and I wanted the PRD to be concise. I also used Groq for their free API usage as it forced me to optimize my constraints drastically.

## Choice to modify and export

Every line can be selected and commented on, for the AI to make revisions to it. This will only affect the independent section of the PRD document, ensuring we don't burn tokens generating the whole thing again. Same goes for system design doc, test doc, and sprint backlog document.

We can export PRD in a cleanly formatted PDF with every section starting on a new page, export as markdown for raw changes and even edit inline for quick changes. All changes autosave so user does not need to worry about losing their changes.

# System design document:

The system design document has the following sections, taken from the context from PRD itself.

Overview & context
Goals & non-goals
Architecture overview
Key components & responsibilities
Data model
APIs & interfaces
Data flow & sequences
Scalability & performance
Reliability & availability
Security & privacy
Observability
Deployment & infrastructure
Trade-offs, alternatives & risks

I chose these fields as to align the team on scope and non-goals and how it lays out the architecture and components, defines the data model, and spells out APIs and end-to-end flows. It also answers scale, failure handling, security, observability, and deployment.

The tab next to it also contains test specs.

# Sprint backlog

Output is user stories in a table (Atlassian-style), grounded in PRD requirements and system design document. It will provide:

Sprint goal: one clear outcome for the sprint (MVP deliverable)
Sprint details: duration (e.g. 2 weeks), team size/capacity, roles
Backlog items table: user stories derived from PRD reqs (ID, story, epic, priority, story points, status, assignee)
Epics overview: grouped work themes with story count and total points
Definition of Done: checklist for when a story/sprint is truly finished
Dependencies & risks: blockers, external teams, technical risks + mitigations
Retrospective template: what went well / improve / action items (filled after sprint)
